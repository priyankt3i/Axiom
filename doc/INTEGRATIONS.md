# System Integrations & Setup Guide

This guide details how to configure Hermes Multi-Agent Orchestrator to integrate with your existing Application Lifecycle Management (ALM), Source Control, and CI/CD tools. 

The configuration for these systems is managed through the **Settings & Integrations** panel in the platform UI.

## 1. Source Control (GitHub / GitLab)

Hermes requires access to your source control to checkout branches, commit code generated by the AI Developer Agent, and open Pull Requests.

### GitHub Integration
1. Navigate to **Settings & Integrations** -> **Source Control**.
2. Click **Connect GitHub App**.
3. You will be redirected to the GitHub OAuth flow to install the "Hermes AI Orchestrator" GitHub App onto your organization or repository.
4. Once authorized, Hermes will receive an installation token used to perform Git operations on your behalf.

### GitLab Integration
1. Navigate to **Settings & Integrations** -> **Source Control**.
2. Click **Configure Webhook** under GitLab.
3. Provide your GitLab Personal Access Token or Project Access Token (needs `api` and `write_repository` scopes).
4. Enter your GitLab instance URL (if self-hosted).

## 2. Issue Tracking (ALM) - Jira / Linear

Integrate with ALM systems to automatically ingest stories, bugs, and tasks into the Hermes Dispatcher.

### Jira Software
1. Go to **Settings & Integrations** -> **Issue Tracking (ALM)**.
2. Click **Setup OAuth 2.0**.
3. Authorize Hermes in your Atlassian workspace.
4. Select the specific Jira Projects and Issue Types (e.g., "Story", "Task") that should be automatically synced to the Hermes Ingestion queue.
5. Hermes will populate the Dispatcher task parameters using the Jira issue description and acceptance criteria.

### Linear
1. In the Linear app, navigate to Settings -> API -> Personal API Keys and generate a new key.
2. In Hermes **Settings & Integrations**, click **Add API Key** under Linear.
3. Paste the API key. 
4. Configure a specific Linear label (e.g., `ai-ready`) that will trigger Hermes ingestion.

## 3. CI/CD & Deployments

While Hermes runs a sandboxed Kubernetes verification loop natively, it should trigger your actual CI/CD pipelines once a Pull Request is approved and merged.

### Jenkins Integration
1. Go to **Settings & Integrations** -> **CI/CD & Deployments**.
2. Click **Configure Webhook**.
3. Provide your Jenkins URL and an API token.
4. Hermes will automatically trigger downstream builds by calling the Jenkins REST API after a successful Proof of Work phase.

## 4. User & Role Management

Hermes utilizes Role-Based Access Control (RBAC) to ensure security and auditability. 
Navigate to **Settings & Integrations** -> **User & Role Management** to manage access.

### Available Roles
* **Platform Admin**: Full access to all system settings, billing (Ledger), and agent configurations.
* **Manager**: Can approve or rollback Pull Requests, view the ledger, and dispatch new tasks.
* **Business Analyst (BA)**: Can ingest and dispatch tasks, view pipeline status, but cannot approve PRs or view financial ledgers.

### Inviting Users
1. Click **Invite User**.
2. Enter their email address.
3. Select their Role from the dropdown.
4. They will receive an email invitation to join the workspace.

## 5. System Status Indicators

At the bottom right of the Hermes UI, you will see two status indicators:

* **TEMPORAL CLUSTER SECURED**: This indicates the health of the connection between the Node.js backend and the Temporal workflow orchestrator. If this turns red, long-running agent loops cannot be processed.
* **K8S VERIFICATION POD ONLINE**: This indicates whether the ephemeral Kubernetes namespace pool is available to spin up sandbox environments for the QA Agent.
